GOVERNANCE
==========

This project follows a lightweight governance model to keep decisions
transparent and predictable. The document below is a starter template —
adjust roles and processes as the project grows.

Roles
- Project Lead: Kate (final decision authority for major changes)
- Maintainers: individuals with write access who review and merge PRs
- Contributors: anyone submitting issues or pull requests

Decision Process
- Small changes: merged after 1 maintainer approval and passing CI.
- Medium changes: require 2 maintainer approvals or consensus in issues.
- Large/controversial changes (API, license, major design): discuss in an issue and seek majority approval from maintainers; the Project
  Lead has final say.

Release Policy
- Use semantic versioning for releases.
- Tag releases in Git and publish change notes to `CHANGELOG.md`.

Code of Conduct
- Adopt a standard Code of Conduct (add `CODE_OF_CONDUCT.md`), and enforce respectful collaboration.

Licensing
- Code: Apache-2.0
- Public cases: CC BY-NC 4.0

How to propose changes
1. Open an issue describing the problem and proposed approach.
2. Create a branch and a PR referencing the issue.
3. Include tests and documentation for the change.

Security
- For security issues, contact maintainers privately (email listed in [`README.md`](./README.md)) before opening a public issue.
